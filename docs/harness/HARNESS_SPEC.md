# OpenClaw 可执行规范：Bootstrap / Checklist

这份文档不是理念说明，而是执行规范。

## 一、Bootstrap 规范

### 1. 触发时机
以下任务开始前，必须先做 bootstrap：
- 调研类
- 文档类
- 代码类
- 工作流类
- 需要 subagent 的复杂任务

### 2. 必须注入的上下文
每次任务开始时，至少明确这 7 项：

1. **当前工作目录**
   - 默认：`/root/.openclaw/workspace`

2. **当前通道 / 会话类型**
   - 当前是否 Feishu
   - 当前是否 direct chat
   - 当前是否 group

3. **当前可写位置**
   - 本地 workspace
   - Feishu Doc / Drive
   - 哪个位置是最终沉淀位置

4. **默认工具策略**
   - 搜索优先：Tavily / 现有搜索工具
   - 文档优先：Feishu Doc / Drive
   - 有一等工具时，不绕 CLI

5. **当前模型与路由常识**
   - 主模型是什么
   - 是否需要强模型 / 便宜模型 / subagent

6. **用户偏好**
   - 语言：简体中文
   - 风格：简洁清晰
   - 重要内容优先沉淀到飞书

7. **任务类型模板**
   - 当前任务属于：调研 / 文档 / 代码 / 工作流 中的哪一类
   - 对应采用哪套流程

### 3. 各任务类型的固定启动问题

#### 调研类
开始前必须明确：
- 用户真正要回答的问题是什么
- 需要读原文还是二手总结即可
- 最终输出到聊天、飞书文档，还是两者都要

#### 文档类
开始前必须明确：
- 写到哪个 doc / folder
- 是新建还是更新
- 目标是长文档、精简版，还是结构化摘要

#### 代码类
开始前必须明确：
- 项目根目录在哪
- 要改哪些文件
- 验证命令是什么
- 完成标准是什么

#### 工作流类
开始前必须明确：
- 是否多步骤
- 是否需要 subagent
- 最终由谁汇总
- 最终产物落在哪里

## 二、Task Checklist 规范

### 1. 通用完成门槛
任何任务结束前，都必须回答这 5 个问题：

1. 需求真的完成了吗？
2. 输出是否符合用户要求？
3. 是否遗漏了显式要求？
4. 是否需要写入文档 / 飞书？
5. 是否有证据证明结果成立？

只要有一项回答不了，就不能视为“完成”。

### 2. 调研类 Checklist
结束前必须检查：
- 是否至少读了核心来源 / 原文
- 是否区分了事实与判断
- 是否提炼成结构化结论
- 是否明确回答“这对用户意味着什么”
- 如果需要存档，是否已经写入飞书

### 3. 文档类 Checklist
结束前必须检查：
- 是否写到了正确 doc / folder
- 命名是否正确
- 内容是否完整
- 写入是否成功
- 是否把最终链接 / 结果告知用户

### 4. 代码类 Checklist
结束前必须检查：
- 是否真正改到了目标文件
- 是否运行了验证 / 测试 / 构建
- 是否说明了验证结果
- 是否引入新错误
- 是否给出下一步或风险说明

### 5. 工作流类 Checklist
结束前必须检查：
- 是否所有子任务都收齐
- 是否做了汇总而不是简单拼接
- 是否交叉验证关键结论
- 是否产出了最终可交付结果
- 是否落到用户指定位置

## 三、执行硬规则

### 1. 没 bootstrap，不开始
### 2. 没 checklist，不结束
### 3. 没证据，不算完成
### 4. 没落到指定位置，不算完成
### 5. 多步骤任务默认要 plan / todo

- `AGENTS.md`：保留高层原则
